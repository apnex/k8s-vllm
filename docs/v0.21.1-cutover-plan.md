# v0.21.1 cutover plan

**Status:** waiting on upstream release.\
**Trigger:** `vllm/vllm-openai:v0.21.1` published to Docker Hub.\
**Estimate:** 2026-05-31 to 2026-06-07 at vLLM cadence (rc0 already tagged 2026-05-24).\
**Owner:** main agent.\
**Cross-refs:** [`audit/v0.21.0/CONSOLIDATED.md`](../audit/v0.21.0/CONSOLIDATED.md), [`docs/model-config-matrix.md`](./model-config-matrix.md), [`docs/perf-hypothesis-ledger.md`](./perf-hypothesis-ledger.md), [`docs/model-research-2026-05-24.md`](./model-research-2026-05-24.md).

## TL;DR

The v0.21.1 cutover is NOT just a version bump.\
It potentially unblocks **7 parked investigation threads** plus enables **3 new Tier S model candidates** discovered in the 2026-05 research sweep.\
The plan below sequences a delta-audit → staging smoke-test → cutover → parked-thread re-evaluation → new-candidate bench.

## What changed since v0.20.2 that we care about

Per the v0.21.0 audit + release notes:

- **CUTLASS sm_120 FP8 block-scaled paths added** (PR #41215 — but #43367 reports it still fails on sm_121; sm_120 status uncertain until tested)
- **Improved FlashInfer FP8 support** (deferred cubin mechanism, autotune rework)
- **FP8 FlashInfer autotune temporarily disabled for correctness** in v0.21.0 — caveat
- **TOKENSPEED_MLA backend** for Blackwell (MLA architectures only — doesn't help our Qwen3-Coder)
- **`VLLM_USE_FLASHINFER_SAMPLER` default flip** to `True` — mitigation: set to `0` explicitly
- **Qwen3CoderTool required-tool-choice fix** (PR #42292) — lands in v0.21.1 specifically
- **Core routing-replay revert** (PR #42434) — v0.21.0 shipped the buggy variant; v0.21.1 reverts

Dispatch an Opus subagent with this brief:

1. Diff v0.21.0..v0.21.1 (small — expect ≤30 PRs since rc0).
2. **Specifically audit FP8 / FlashInfer / sm_120 path changes** v0.20.2 → v0.21.1.
3. For each of our parked threads (H3, H7, H14, R15, R6/R7, G14, G18), state explicitly whether v0.21.1 changes the gating reason.

Output: `audit/v0.21.1/CONSOLIDATED.md` mirroring the v0.21.0 audit structure.\

## Phase 2 — Staging smoke test

Pull `vllm/vllm-openai:v0.21.1` into k3s containerd without disturbing the running v0.20.2 pod:
```bash
docker pull vllm/vllm-openai:v0.21.1
docker save vllm/vllm-openai:v0.21.1 | sudo k3s ctr images import -
```

Apply mitigations identified by the v0.21.0 audit:
- `VLLM_USE_FLASHINFER_SAMPLER=0` in container env (until soaked)
- `--fingerprint-mode=none` (cosmetic; avoids response-shape change for downstream clients)
- Set `top_k=50` watchdog mitigation if startup hang (#42987) reproduces

Bring up a parallel test deployment in a separate namespace (`vllm-staging`) with its own LoadBalancer service from `vllm-pool` if a second IP is added, OR via `kubectl port-forward`.\
**Do NOT replace the production deployment yet** — the soak in progress is the primary signal stream.

Run the standard regression battery on the staging deployment:
- `tools/perf-snapshot.sh --label v0.21.1-staging` (compare against R11 baseline + R-cutover from 2026-05-24)
- `tools/polyglot-bench.sh 5 python` (compare pass/fail pattern)

## Phase 3 — Cutover decision gate

Promote v0.21.1 to production if ALL hold:

1. Perf within ±5% of R11 baseline (no decode regression)
2. Polyglot pass-rate in-distribution with R11 (3-4/5 acceptable)
3. No startup hang on first launch
4. No engine wedge in 1-hour sustained traffic test
5. Audit-delta found no Tier S regression specific to our config

If yes:
```bash
# in k8s/deployment.yaml — bump image tag, re-apply
sed -i 's/vllm-openai:v0.20.2/vllm-openai:v0.21.1/' k8s/deployment.yaml
kubectl apply -f k8s/deployment.yaml
```

Restart pauses soak metrics for ~5 min; otherwise continuous.\
Bump tag in `k8s/configmap.yaml`'s app-version label too.

If no on any gate: stay on v0.20.2; re-poll on v0.21.2 or next release.

## Why this is a "real cutover" not a "tag bump"

The v0.21.0 audit established the **decision** to skip v0.21.0; this plan establishes the **opportunity** in v0.21.1.\
Two parked investigation threads (H3, H14) plus one parked model (R15) plus a NVFP4 weight-loader bug (R6/R7) plus a strategic FP8 weights path (H7) plus two new Tier S candidates (Qwopus-27B-FP8, GLM-4.7-Flash-REAP) all converge on the same upstream version.\
Treating this as a project milestone — not just a routine version bump — sequences the work so we extract maximum signal in the post-cutover days.
