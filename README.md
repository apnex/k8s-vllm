# vLLM on AORUS RTX 5090 eGPU (Fedora 43, Docker)

Containerised vLLM OpenAI-compatible API server,
running the patched NVIDIA driver from
[`apnex/aorus-5090-egpu`](https://github.com/apnex/aorus-5090-egpu)
or
[`apnex/nvidia-driver-injector`](https://github.com/apnex/nvidia-driver-injector).
This repo is everything that depends on having a healthy CUDA stack
to start vLLM containers against.

## Why Docker

- Official vLLM container images pin every Python / torch / triton / CUDA version against tested combos.
  Avoids dependency hell.
- vLLM's CI primarily exercises the container path;
  fewer "this combo doesn't work" surprises.
- Clean install / uninstall — no leftover venv files;
  `docker rmi` and we're done.
- vLLM upgrades are a tag change, not a 6-10 GB pip dance.

The kernel-level bugs in NVIDIA's Blackwell-over-Thunderbolt path are NOT fixed by Docker;
they live below the container boundary
(handled by the patched driver in the companion repos).
Docker just simplifies the userspace layer.

## Prerequisites

The host must already have the patched NVIDIA driver loaded — either via
the [`aorus-5090-egpu`](https://github.com/apnex/aorus-5090-egpu) systemd stack
or the [`nvidia-driver-injector`](https://github.com/apnex/nvidia-driver-injector) container.
Verify with:

```bash
cat /sys/module/nvidia/version       # should print 595.71.05-aorus.<n>
nvidia-smi -L                        # should list the RTX 5090
```

Also required, installed by `setup.sh` in this repo:

- `nvidia-container-toolkit` (from NVIDIA's container repo)
- CDI spec at `/etc/cdi/nvidia.yaml` (auto-generated by `nvidia-ctk cdi generate`)
- Docker runtime configured for `nvidia`

## Goal

Serve LLM inference using vLLM's OpenAI-compatible API,
container-native, on the RTX 5090.
Achieved 2026-05-09 with vLLM v0.20.2 + Llama 3.1 8B Instruct
running at ~98 tok/s steady-state decode.

### Bring-up — current path

`docker-compose.yml` is the canonical bring-up:

```bash
docker compose up -d
docker compose logs -f             # watch model load (first run downloads ~16 GB weights)
curl -s http://127.0.0.1:8000/v1/models | jq
```

See the `## Smoke test` section below for an end-to-end validation curl.

### History — May 1 attempt and what unblocked it

Initial vLLM 0.20.0 bring-up on 2026-05-01 hit 8+ host freezes across both Docker and native installs.
Failure triggers:

1. `torch.distributed.init_process_group(backend='nccl', ...)` — physically took the eGPU off PCIe.
2. Post-model-load profile run — second hard freeze even with NCCL preinit-via-gloo workaround.

Both triggers were *driver-layer* failures specific to Blackwell-on-Thunderbolt.
Resolution arrived 2026-05-08 via the in-driver patches at
[`apnex/aorus-5090-egpu`](https://github.com/apnex/aorus-5090-egpu):

- **Lever Q-watchdog** detects Mode B wedges and DISCONNECTs the device cleanly instead of hanging.
- **Lever M-recover** registers `pcie_err_handlers` that bus-reset on `rm_init` failure.
- **iommu=off (Lever T)** eliminates DMAR fault GSP_LOCKDOWN cascades.
- **H17 LnkCtl2 cap** holds the bridge at Gen1 before `nvidia` binds.
- **Patches 0029/0030** mitigate the close-path bug class.

With those in place, vLLM 0.20.2 brings up cleanly on this hardware (validated 2026-05-09).
Both prior freeze triggers passed without host disturbance:
NCCL init succeeded silently; post-load profiling/warmup ran in 0.44s.

The May 1 attempt evidence is preserved in `archive/vllm-attempts-2026-05-01/`.
The native-install tooling (`setup-native-vllm-venv.sh`, `tty-native-vllm-test.sh`,
`vllm-gloo-preinit.py`) is retained in `tools/` as a fallback path
but the Docker route is the supported one going forward.

## Files installed on host

After running `setup.sh`:

| Path | Purpose |
|---|---|
| `/etc/yum.repos.d/nvidia-container-toolkit.repo` | NVIDIA's container toolkit dnf repo |
| `nvidia-container-toolkit` package + deps | container GPU access tooling |
| `/etc/cdi/nvidia.yaml` | Container Device Interface spec (auto-generated by `nvidia-ctk cdi generate`) |
| `/etc/docker/daemon.json` | docker runtime configured for `nvidia` (preserves existing keys) |

Things explicitly NOT created by `setup.sh`:

- The vLLM container itself (managed by `docker-compose.yml` in this repo).
- Model weights — those download on first use into `/root/.cache/huggingface/`.

## Smoke test

After `docker compose up -d`, wait for the container to report `(healthy)` —
cold start is ~80s with weights cached, ~5 min on first run while weights download.

```bash
# Container healthy?
docker compose ps                                   # STATUS column should say (healthy)

# Model registered?
curl -s http://127.0.0.1:8000/v1/models | jq -r '.data[].id'
# unsloth/Meta-Llama-3.1-8B-Instruct

# One-shot inference
curl -s http://127.0.0.1:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{
      "model": "unsloth/Meta-Llama-3.1-8B-Instruct",
      "messages": [{"role": "user", "content": "Say HELLO once."}],
      "max_tokens": 16,
      "temperature": 0
    }' | jq -r '.choices[0].message.content'
# HELLO.
```

## Performance baseline (2026-05-09)

Two configurations measured on this hardware
(vLLM v0.20.2, max_model_len=32768, gpu_memory_utilization=0.9,
no `--enforce-eager` — torch.compile + CUDAGraphs on).

### Llama 3.1 8B Instruct (initial bring-up)

| Metric | Value |
|---|---|
| Architecture | Dense, 8B params, all active per token |
| Quantisation | bf16 |
| Weights size | ~16 GB |
| Cold start (weights cached) | ~80s |
| Weight load to VRAM | 5.98s (TB4-saturated, ~2.66 GB/s) |
| Steady-state decode (250 tok ×3) | 97.6 / 98.8 / 98.8 tok/s |
| VRAM at idle | ~17.4 GiB / 32.6 GiB |
| OpenCode tool-call reliability | mediocre — 2-3 retries on JSON schema |

### Gemma 4 26B-A4B (current default)

Released 2026-05-07; AWQ-4bit via `cyankiwi/gemma-4-26B-A4B-it-AWQ-4bit`.

| Metric | Value |
|---|---|
| Architecture | **MoE**, 26B total / **4B active per token** |
| Quantisation | AWQ-4bit (Marlin kernel for WNA16 MoE) |
| Weights size | 16.0 GB |
| Cold start (weights cached) | ~110s (Triton attention + larger compile graph) |
| Weight load to VRAM | 40.6s (AWQ dequant during load) |
| Steady-state decode (250 tok ×3) | **222.7 / 223.8 / 223.8 tok/s** |
| VRAM at idle | 16.47 GiB / 32.6 GiB |
| Multimodal | yes (vision + video; not exercised by default) |
| OpenCode tool-call reliability | clean first-shot; native `gemma4` parser |

The MoE architecture is the win:
26B reasoning capacity with the per-token compute of a 4B model.
Decode is **2.3× faster** than dense Llama 3.1 8B
on the same hardware,
and tool-call accuracy is dramatically better
(zero retries on the OpenCode bash tool vs. several for Llama 3.1 8B).

For comparison,
the previous `ollama` deployment on this hardware reached 105% of WSL2 parity on Llama 3.1 8B
(per `aorus-5090-egpu`'s `feedback_perf_parity_confirmed.md` memory entry,
measured 2026-05-04).
Gemma 4 26B-A4B at ~223 tok/s sets a new ceiling for this stack.

### Switching models

```bash
# Default (Gemma 4 26B-A4B):
docker compose up -d

# Swap to Llama 3.1 8B (or any HF repo) without editing the compose file:
VLLM_MODEL=unsloth/Meta-Llama-3.1-8B-Instruct \
    VLLM_SERVED_NAME=llama-3.1-8b-it \
    docker compose up -d
```

Then update `~/.config/opencode/opencode.json` so the `models` key under `vllm-local`
matches the new `served-model-name`.
